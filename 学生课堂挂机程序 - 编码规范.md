## 一、总体原则

1. **类型安全优先**：严格规避模糊类型使用（禁止使用 `Object`、`<?>` 作为返回类型，仅允许在极端例外场景使用类型转换）
2. **SOLID 原则遵循**
   - 单一职责原则：一个函数/类/组件/模块仅负责一项核心功能
   - 开闭原则：对功能扩展开放，对原有代码修改关闭
   - 依赖倒置原则：高层模块与低层模块均依赖抽象接口，不直接依赖具体实现
   - 接口隔离原则：避免冗余接口定义，不强迫使用者依赖无需使用的功能
3. **模块化设计**：合理拆分功能单元，降低模块耦合度，提升复用性
4. **中文注释统一**：所有注释均使用中文编写，聚焦「为什么做」而非「做了什么」
5. **简洁命名原则**：命名表意清晰，避免冗余前缀/后缀，超长单词可使用合理缩写
6. **统一代码格式**：使用 Google Java Format 格式化代码，统一缩进规则（4空格缩进）
7. **数据/资源一致性**：保障数据类型匹配、字段统一，避免数据冗余；资源管理遵循「申请即绑定释放」原则
8. **独立模块文档**：独立功能模块必须配套 README.md 自述文件，修改/扩展前需先阅读文档
9. **代码自审查**：开发完成后自行审查代码，确保符合本规范后再提交评审
10. **临时代码清理**：临时测试代码使用完毕后立即删除，同步清理编译生成的冗余文件
11. **冗余代码剔除**：减少代码堆砌，优先封装公共逻辑，删除无用代码、注释与依赖
12. **最佳实践优先**：遵循「如无必要，勿增实体」原则，优先使用成熟方案避免重复造轮子，优先使用成熟的开源库或第三方库
13. **注意可扩展性，尽量避免使用布尔类型作为参数**

## 三、类型定义规范

### 3.1 通用类型要求

1. **优先使用项目语言支持的类型定义方式**：使用接口（interface）、实体类（class）、枚举（enum），明确数据结构
2. **禁止滥用模糊类型**：
   - 禁止使用 `Object` 作为返回类型（仅在极端例外场景使用）
   - 禁止使用 `<?>` 作为返回类型
   - 项目内已定义类型必须严格引用（如 `WakeWordResult`、`DetectionState`）
3. **对外部输入数据进行类型与格式校验**：接口请求、用户输入、文件读取等必须进行校验
4. **复杂数据结构需单独提取为公共类型**：避免重复定义，使用 DTO、VO、Entity 等模式

### 3.2 类型校验与守卫

1. **使用项目语言支持的方式实现类型判断/断言**：
   - 使用 `Objects.requireNonNull()` 检查 null
   - 使用 `instanceof` 进行类型检查
   - 使用 `Class.cast()` 进行类型转换（仅在确保安全时使用）
2. **自定义类型校验函数需明确返回值**：确保类型合法性可追溯
3. **枚举类型优先使用语言原生枚举**：避免魔法常量（如 `String` 或 `int` 状态码）

### 3.3 语言特有类型配置（Java）

| 项目/语言类型 | 类型定义方式 | 示例 | 备注 |
|--------------|-------------|------|------|
| **Java 项目** | 接口（interface） | `public interface AudioListener { void onAudioReady(byte[] data); }` | 定义契约 |
| **Java 项目** | 实体类（class） | `public class WakeWordDetector { ... }` | 具体实现 |
| **Java 项目** | 数据传输对象（DTO） | `public class RecognitionResult { private String text; ... }` | 数据传输 |
| **Java 项目** | 值对象（Value Object） | `public record AudioConfig(int sampleRate, int channels) {}` | 不可变数据（Java 16+） |
| **Java 项目** | 枚举（enum） | `public enum ModelType { WAKE_WORD, VAD, ASR }` | 类型安全常量 |
| **Java 项目** | 泛型类型 | `public interface Service<T> { T execute(); }` | 类型参数化 |

## 四、注释规范

### 4.1 通用注释要求

1. **注释语言**：统一使用中文，排版整洁，与代码保持合理缩进
2. **必加注释场景**：
   - 公共函数/类/接口
   - 复杂业务逻辑
   - 特殊处理逻辑（如算法优化、异常处理）
   - 不易理解的代码片段
3. **注释优先级**：先解释「业务背景/设计考量」，再描述「功能实现」
4. **禁止注释**：
   - 无用代码注释（如 `// TODO: 删除`）
   - 重复代码描述注释（如 `// 返回 true`）
   - 显而易见的逻辑注释（如 `i++`）

### 4.2 文档注释规范（Javadoc）

**公共函数/类/接口必须添加完整文档注释**，包含核心信息：功能描述、参数说明、返回值说明。

**可选补充**：异常说明（@throws）、使用示例（@example）、注意事项（@note）

## 五、事件/通信系统规范

1. **禁止使用魔法字符串/常量作为通信标识**：优先使用枚举或唯一符号统一管理
2. **跨模块/跨实例通信需使用统一通信机制**：避免直接调用内部方法
3. **通信数据需定义明确类型**：保障数据一致性
4. **通信后需处理异常场景**：避免消息丢失或处理失败导致的状态异常
5. **特有通信工具/框架**：
   - **JavaFX 事件机制**：使用 JavaFX `EventHandler<T>` 和自定义 `Event`

   - **观察者模式**：使用 JavaFX `Property` 和 `ChangeListener`

## 八、错误处理规范

### 8.1 通用错误处理要求

1. **统一日志工具**：禁止使用 `System.out` 或 `System.err`，统一使用 SLF4J + Logback
2. **日志等级划分**：
   - `ERROR`：严重错误，影响系统运行
   - `WARN`：警告信息，可能影响功能
   - `INFO`：关键业务操作信息
   - `DEBUG`：调试信息（生产环境关闭）
3. **安全数据处理**：对 JSON/配置文件等外部数据使用安全解析函数，避免解析失败导致程序崩溃
4. **异常捕获**：所有可能抛出异常的代码块必须添加捕获逻辑，避免程序异常退出
5. **错误反馈**：异常信息需包含关键上下文（如参数、调用链），便于问题排查

### 8.2 特有日志/异常工具（Java）

| 项目/语言类型 | 日志工具 | 异常处理工具 | 备注 |
|--------------|---------|------------|------|
| **Java 项目** | SLF4J + Logback | 自定义异常体系 | 日志配置文件：`logback.xml` |

## 九、状态/资源管理规范

### 9.1 通用管理要求

1. **禁止滥用全局状态/全局变量**：优先使用局部状态/作用域变量

### 9.4 线程安全示例

```java
public class CircularBuffer {

    private final byte[] buffer;
    private final AtomicInteger writeIndex = new AtomicInteger(0);
    private final AtomicInteger readIndex = new AtomicInteger(0);
    private final AtomicReference<Thread> writeThread = new AtomicReference<>();

    public void write(byte[] data) {
        Thread currentThread = Thread.currentThread();
        if (!writeThread.compareAndSet(null, currentThread)) {
            throw new IllegalStateException("Only one thread can write at a time");
        }

        try {
            for (byte b : data) {
                buffer[writeIndex.getAndIncrement() % buffer.length] = b;
            }
        } finally {
            writeThread.set(null);
        }
    }
}
```

```java
/**
 * 音频格式转换工具类
 *
 * <p>提供音频数据格式转换功能，包括 PCM 转换为浮点数、采样率转换等。
 */
public final class AudioUtils {

    private AudioUtils() {
        // 防止实例化
    }

    /**
     * 将 16-bit PCM 数据转换为浮点数数组
     *
     * @param pcmData 16-bit PCM 数据，小端序
     * @return 归一化到 [-1.0, 1.0] 的浮点数数组
     */
    public static float[] pcmToFloat(byte[] pcmData) {
        if (pcmData == null) {
            throw new IllegalArgumentException("PCM data cannot be null");
        }

        float[] floatData = new float[pcmData.length / 2];
        for (int i = 0; i < floatData.length; i++) {
            int sample = (pcmData[2 * i] & 0xFF) | (pcmData[2 * i + 1] << 8);
            floatData[i] = sample / 32768.0f;
        }
        return floatData;
    }

    /**
     * 计算音频数据的 RMS（均方根）音量
     *
     * @param audioData 音频数据
     * @return RMS 值（0.0 - 1.0）
     */
    public static double calculateRMS(float[] audioData) {
        if (audioData == null || audioData.length == 0) {
            return 0.0;
        }

        double sum = 0.0;
        for (float sample : audioData) {
            sum += sample * sample;
        }
        return Math.sqrt(sum / audioData.length);
    }
}
```
